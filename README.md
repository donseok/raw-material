# 철스크랩 수급 계획 및 구매 관리 시스템 문서

## 1. 한 줄 요약

이 프로젝트는 철스크랩 구매 업무를 웹 화면으로 시연하는 정적 대시보드형 프로그램입니다.  
수급계획, 공장배분, 발주, 거래처, 수입계약, 재고, 구매실적을 탭으로 나눠 보여주며, 일부 화면은 직접 수정 후 브라우저에 저장할 수 있습니다.

## 2. 이 문서가 필요한 사람

- 개발자: 파일 구조, 상태 관리 방식, 실제 동작 범위를 빠르게 파악할 수 있습니다.
- 설계자: 화면 패턴, 정보 구조, UI 톤앤매너를 한 번에 이해할 수 있습니다.
- 기획자: 현재 구현 범위와 운영 시스템으로 가기 위한 보완 지점을 구분해서 볼 수 있습니다.

## 3. 실행 방법

- 진입 파일: `index.html`
- 실행 방식: 브라우저에서 `index.html`을 직접 열면 됩니다.
- 참고: 차트는 `Chart.js` CDN을 사용하므로, 차트 표시에는 인터넷 연결이 필요합니다.

## 4. 현재 코드 구조

이 프로젝트는 단일 HTML이 아니라, 정적 파일을 역할별로 나눈 구조입니다.

```text
index.html
assets/
css/
  variables.css
  layout.css
  components.css
js/
  utils.js
  data.js
  plan.js
  alloc.js
  order.js
  supplier.js
  import.js
  inventory.js
  stats.js
  app.js
```

### 파일별 역할

| 파일 | 역할 |
| --- | --- |
| `index.html` | 전체 화면 마크업과 탭 구조 |
| `css/variables.css` | 색상, 그림자 등 공통 디자인 변수 |
| `css/layout.css` | 헤더, 탭, 그리드, 카드 등 전체 레이아웃 |
| `css/components.css` | 표, 배지, 버튼, 모달, 차트 등 컴포넌트 스타일 |
| `js/utils.js` | 공통 함수, 숫자 포맷, 토스트, 셀 편집, 차트 정리 |
| `js/data.js` | 기본 데이터와 화면용 샘플 데이터 |
| `js/plan.js` | 수급계획 화면 로직 |
| `js/alloc.js` | 공장배분 화면 로직 |
| `js/order.js` | 발주관리 화면 로직 |
| `js/supplier.js` | 거래처관리 화면 로직 |
| `js/import.js` | 수입계약 화면 로직 |
| `js/inventory.js` | 재고현황 화면 로직 |
| `js/stats.js` | 구매실적 화면 로직 |
| `js/app.js` | 초기화, 탭 이동, 공통 이벤트 연결 |

## 5. 프로그램이 하는 일

이 시스템은 아래 7개 업무 영역을 한 화면군에서 보여줍니다.

| 탭 | 목적 | 현재 구현 수준 |
| --- | --- | --- |
| 수급계획 | 월별 계획량, 국산/수입 비율, 실적 관리 | 일부 실사용 가능 |
| 공장배분 | 인천/포항 공장별 계획/실적 배분 관리 | 일부 실사용 가능 |
| 발주관리 | 발주 목록 조회, 신규 발주 등록, 발주서 출력 | 부분 실사용 가능 |
| 거래처관리 | 거래처 현황 및 납품 비중 확인 | 기본 입력 가능 |
| 수입계약 | 수입 계약 현황과 진행 상태 확인 | 기본 입력 가능 |
| 재고현황 | 공장별/등급별 재고 현황 확인 | 조회 중심 |
| 구매실적 | 구매량/금액 통계, CSV 출력 | 조회 중심 |

## 6. 화면별 상세 설명

### 6.1 수급계획

- 월별 계획, 국산 물량, 수입 물량, 실적을 관리하는 화면입니다.
- KPI 카드, 비율 슬라이더, 막대 차트, 편집 가능한 표로 구성됩니다.
- 사용자가 할 수 있는 일:
  - 셀 더블클릭 편집
  - 엑셀 데이터 `Ctrl + V` 붙여넣기
  - 초기화
  - 브라우저 저장값 재사용
- 저장 방식:
  - `localStorage.planData`
- 구현상 주의:
  - `계획 저장` 버튼은 실제 저장이 아니라 안내용 `alert`만 띄웁니다.
  - `이전 계획 불러오기` 버튼은 동작이 없습니다.
  - 슬라이더는 표시값만 바꾸고 표 데이터까지 자동 재계산하지는 않습니다.

### 6.2 공장배분

- 인천공장/포항공장의 계획량과 실적을 관리하는 화면입니다.
- KPI 카드, 진행률 바, 도넛 차트, 편집 가능한 표로 구성됩니다.
- 사용자가 할 수 있는 일:
  - 셀 더블클릭 편집
  - 엑셀 데이터 붙여넣기
  - 초기화
- 저장 방식:
  - `localStorage.allocData`

### 6.3 발주관리

- 발주 목록 조회, 신규 발주 모달, PDF 발주서 미리보기 기능을 제공합니다.
- 실제 구현은 데모 성격이 강합니다.
- 현재 동작:
  - 발주 목록 표시
  - 신규 발주 모달 열기
  - 입력값 기준으로 새 발주 행 추가
  - 브라우저 저장 후 재접속 시 유지
  - PDF 발주서 미리보기 모달 열기
- 한계:
  - 서버 저장은 없습니다.
  - 실제 PDF 파일 생성이 아니라 인쇄/PDF 저장 화면을 여는 방식입니다.

### 6.4 거래처관리

- 거래처 테이블과 납품 비중/추이 차트를 보여줍니다.
- 기본 데이터는 `js/data.js`에 있고, 신규 거래처는 브라우저에 저장됩니다.
- 등록된 거래처는 표와 비중 차트에 바로 반영됩니다.

### 6.5 수입계약

- 국가, 공급사, 등급, 물량, CFR, 환율, 선적일, 도착예정일, 상태를 조회합니다.
- 상태별 계약 건수 차트를 제공합니다.
- 신규 계약 모달 입력값으로 목록과 KPI, 차트를 갱신합니다.
- 등록된 계약은 브라우저에 저장됩니다.

### 6.6 재고현황

- 공장별/등급별 현재 재고와 적정 재고를 비교합니다.
- KPI 카드, 공장별 막대 차트, 상세 표로 구성됩니다.
- 조회 전용 화면입니다.

### 6.7 구매실적

- 월별 구매 실적 통계를 보여줍니다.
- 월별 구매량/금액 차트와 등급별 구매 비중 차트를 제공합니다.
- 사용자가 할 수 있는 일:
  - CSV 다운로드
  - 인쇄
- 데이터는 고정 배열 기반입니다.

## 7. 데이터와 상태 관리 방식

현재 데이터는 크게 3종류입니다.

### 7.1 브라우저에 저장되는 데이터

이 데이터는 수정 후 다시 열어도 유지됩니다.

```js
planData = [{ plan, dom, imp, actual }]
allocData = [{ ic, ia, pc, pa }]
orders = [{ no, sup, grade, qty, price, plant, date, status }]
suppliers = [{ code, name, region, rep, tel, cap, ytd, grade, rate }]
imports = [{ no, country, sup, grade, qty, cfr, fx, ship, eta, status }]
```

- `planData`
  - `plan`: 계획 물량
  - `dom`: 국산 물량
  - `imp`: 수입 물량
  - `actual`: 실적 물량
- `allocData`
  - `ic`: 인천 계획
  - `ia`: 인천 실적
  - `pc`: 포항 계획
  - `pa`: 포항 실적
- `orders`
  - 발주 목록과 PDF 미리보기 기준 데이터
- `suppliers`
  - 거래처 목록과 거래처 비중 차트 데이터
- `imports`
  - 수입계약 목록, KPI, 상태 차트 데이터

### 7.2 메모리에만 있는 화면용 샘플 데이터

아래 데이터는 `js/data.js`의 기본값을 사용하며, 현재는 조회용입니다.

- `inventory`
- `statsData`

### 7.3 화면 계산값

저장값이 아니라 렌더링 시 매번 다시 계산되는 값입니다.

- KPI 카드 수치
- 달성률
- 진행률 바
- 차트 데이터
- 합계 행

## 8. 동작 흐름

### 8.1 초기 진입

1. `index.html`이 CSS와 JS 파일을 순서대로 불러옵니다.
2. `js/app.js`가 현재 날짜를 표시합니다.
3. `loadFromStorage()`가 `planData`, `allocData`를 읽습니다.
4. `initOrders()`, `initSuppliers()`, `initImports()`, `initInventory()`, `initStats()`가 조회성 화면을 채웁니다.
5. `renderPlanUI()`, `renderAllocUI()`, `renderPlanChart()`가 초기 화면을 그립니다.

### 8.2 탭 이동

1. 탭 버튼 클릭
2. 활성 탭 CSS 변경
3. `renderCharts()`가 현재 탭 차트를 다시 렌더링

### 8.3 수급계획/공장배분 편집

1. 사용자가 표를 더블클릭하거나 엑셀 데이터를 붙여넣음
2. 숫자 변환 및 검증
3. 전역 데이터 갱신
4. `localStorage` 저장
5. KPI, 표, 차트 재렌더링

## 9. 개발자가 알아야 할 핵심 포인트

- 프레임워크 없는 순수 HTML/CSS/JavaScript 구조입니다.
- 파일은 분리돼 있지만, ES Module이 아니라 전역 함수/변수 공유 방식입니다.
- 따라서 스크립트 로딩 순서가 중요합니다.
- 화면 상태는 전역 배열과 DOM 재렌더링으로 관리합니다.
- 차트는 탭 전환 시 생성/파괴 방식으로 관리합니다.
- 작은 프로토타입에는 빠르지만, 운영 시스템으로 가려면 아래 분리가 필요합니다.
  - API/저장 계층
  - 도메인 모델
  - 모달 폼 처리
  - 검증 로직
  - 공통 렌더링 유틸 또는 컴포넌트 구조

## 10. 설계자가 알아야 할 핵심 포인트

- 전체 UI는 같은 패턴을 반복합니다.
  - 상단 헤더
  - 탭 네비게이션
  - KPI 카드
  - 카드형 섹션
  - 표 + 차트 + 모달
- 색상 체계는 `남색 + 주황` 중심입니다.
- 상태 표현은 초록/주황/빨강으로 명확히 나뉩니다.
- 정보 밀도가 높은 B2B 운영 화면 스타일입니다.
- 모바일 완전 최적화보다는 데스크톱 업무 화면에 더 적합합니다.

## 11. 기획자가 알아야 할 핵심 포인트

현재 시스템은 운영 제품 전 단계의 브라우저 기반 프로토타입에 가깝습니다.

### 실제로 동작하는 영역

- 수급계획 편집/붙여넣기/초기화/브라우저 저장
- 공장배분 편집/붙여넣기/초기화/브라우저 저장
- 발주 등록/브라우저 저장/PDF 미리보기
- 거래처 등록/브라우저 저장
- 수입계약 등록/브라우저 저장
- 차트 조회
- 구매실적 CSV 다운로드
- 발주서 인쇄 화면

### 아직 운영 기능이 아닌 영역

- 사용자 로그인/권한
- 서버 저장
- DB 연동
- 이력 관리
- 승인/결재
- 검색/필터/정렬

## 12. 현재 한계와 리스크

- 실제 업무 데이터와 연결되지 않습니다.
- 저장이 브라우저 `localStorage`에만 되므로, 사용자 환경이 바뀌면 데이터가 유지되지 않습니다.
- 여러 사용자가 동시에 협업할 수 없습니다.
- 입력값 검증이 약합니다.
- 저장이 모두 브라우저 단위라, 사용자 간 공유와 동시 작업이 불가능합니다.
- 전역 변수 중심 구조라 기능이 커질수록 유지보수가 어려워질 수 있습니다.

## 13. 운영 시스템으로 발전시키기 위한 우선순위

### 1단계: 데이터 실체화

- 발주, 거래처, 수입계약, 재고, 실적 데이터를 API와 DB로 분리
- 하드코딩 배열 제거
- 공통 코드 체계와 마스터 데이터 정의

### 2단계: 실제 업무 처리

- 모달 입력값 저장
- 검색/필터/정렬
- 발주 승인 흐름
- 거래처 등록/수정/비활성
- 수입계약 상태 변경 이력

### 3단계: 운영 품질 강화

- 사용자/권한 관리
- 감사 로그
- 입력 검증과 예외 처리
- 엑셀 입출력 고도화
- 화면/로직 분리와 구조 정리

## 14. 결론

이 코드베이스는 철스크랩 구매 업무를 설명하고 시연하기 위한 정적 웹 프로토타입으로 이해하는 것이 가장 정확합니다.  
특히 `수급계획`과 `공장배분`은 직접 수정 가능한 수준까지 구현되어 있고, 나머지 탭은 조회 중심 또는 시연용 화면입니다.  
즉, 현재 문서는 "무엇이 이미 동작하는지"와 "무엇을 다음 단계에서 시스템화해야 하는지"를 빠르게 판단하기 위한 기준 문서입니다.
